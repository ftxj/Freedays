# 地图动画代码架构

## 目标

路线数据分成四层，浏览器呈现作为独立消费端，避免把事实、剪辑、导航和动画插值混成一个不可追溯的数组：

1. **原始 GPS 层**：GPX 及其只读规范化镜像。
2. **人工剪辑层**：剔除规则、手工关键点和故事触发点。
3. **导航道路层**：OSM/OSRM 道路匹配结果。
4. **动画呈现层**：人工几何修正、圆角和插值后的最终路线。

原始 GPX 不直接修改。所有“为了成片”的取舍都留在剪辑指令中，可追踪、可撤销。

各层路径、点来源字段和重新生成命令见 `route_data/README.md`。最终点使用 `source`、`source_id`、`derived_from` 和 `generated` 记录来源。

## 应用五层架构

路线数据的四层来源与网页应用分层是两个维度。浏览器端按以下五层协作：

1. **轨迹数据层**：原始、剔除、导航拟合和人工点，最终编译为 `route-data.js`。
2. **时间线层**：把时间/GPS 指令编译为路线距离事件，只表达“什么时候发生什么”。
3. **播放内核层**：管理距离进度、播放倍率、媒体倍率、完成状态和多原因暂停。
4. **动画渲染层**：根据播放状态更新 MapLibre、路线、角色、镜头、字幕和媒体卡。
5. **UI 层**：播放、重播、倍速、进度条、选点和调试工具；只向播放内核发送命令。

`PlaybackEngine` 使用 `user`、`media`、`sleep` 和 `stop` 等暂停原因，只有所有原因都解除才继续，避免关闭视频时错误恢复用户手动暂停的路线。

## 浏览器模块

| 模块 | 职责 | 不负责 |
|---|---|---|
| `route-model.js` | 距离、时间、速度、转弯角、坐标最近点 | 地图和 UI |
| `directive-engine.js` | 把时间/GPS 指令编译为路线触发点 | 指令显示效果 |
| `timeline-runtime.js` | 标题/媒体事件、跨越触发、seek 重置和手工事件覆盖 | 媒体和字幕样式 |
| `playback-engine.js` | 播放距离、倍率、组合速度因子、暂停原因和完成状态 | MapLibre 和 DOM |
| `playback-controls.js` | 播放、重播、进度、倍速和停靠时长控件 | 路线渲染 |
| `camera-policy.js` | 固定区域、柔和跟随、缩放、俯仰、朝向目标 | 播放进度 |
| `media-overlay.js` | 标题下的照片/视频卡、暂停与恢复、延迟加载 | GPS 定位 |
| `map-animation.js` | MapLibre 图层、角色/镜头渲染和模块编排 | 独立保存播放状态、具体素材清单和轨迹修正数据 |

模块通过 `window.RouteDemo` 暴露接口，不依赖打包工具，仍可用本地 HTTP 服务直接打开。

## 剪辑指令分类

配置文件由 `route-overrides.schema.json` 约束。

`tunnel_ranges` 保存已由 OSM `tunnel=yes` 或人工复核确认的隧道时间区间。播放层会将这些路段压为深红灰色，车辆进入时同步降低亮度；普通 GPS 断点不能自动当作隧道。

### 1. 到达坐标后显示内容

新事件统一写入 `story_events`。触发可以使用 `lonlat` 或 `time`；有可靠 GPS 时优先使用 `lonlat`。

```json
{
  "id": "pass-memory",
  "presentation": "media",
  "lonlat": [90.7719, 29.1409],
  "title": "翻越加若拉山口",
  "subtitle": "海拔 4710 米",
  "video": "photo/example_web.mp4",
  "poster": "photo/example_poster.jpg",
  "priority": "chapter",
  "pause": true
}
```

- `presentation: title`：只显示地点字幕。
- `presentation: media`：显示照片或视频卡。
- `priority: chapter`：重要章节卡。
- `priority: memory`：右下角轻量记忆卡。

静态照片可以在固定比例窗口中做克制的镜头运动。默认仍是静止图，只有显式配置 `photo_motion` 才启用；故事点编辑器中的“静态照片动态化”可以直接预览这些字段。

```json
{
  "id": "lake-photo-memory",
  "presentation": "media",
  "lonlat": [90.7612, 28.8474],
  "title": "湖岸向远方展开",
  "image": "photo/example.jpg",
  "photo_motion": "pan-left-to-right",
  "photo_window_aspect": 1.7777778,
  "photo_motion_duration_ms": 8000,
  "photo_motion_scale": 1.12
}
```

- `photo_motion`：`none`、四向平移、`zoom-in`、`zoom-out` 或两种对角漂移。
- `photo_window_aspect`：固定展示窗口比例，例如 `1.7777778` 为 16:9；不设置则跟随原图。
- `photo_motion_duration_ms`：运动时长；`photo_motion_scale` 控制为裁切预留的缩放，范围为 1–1.5。
- 照片序列可把这些字段写在事件上作为统一默认值，也可写进某个 `clips` 项覆盖单张照片。
- 用户开启“减少动态效果”后，照片自动回到居中静止展示。

旧的 `title_cards`、`supply_events`、`media_events` 继续兼容，可以逐步迁移。

### 2. 删除无叙事价值的折返

使用 `geometry_edits`。起止时间保持不变，只替换中间几何，因此真实时间轴和后续素材不会提前。

```json
{
  "name": "无效往返支线移除",
  "start": "2024-12-22T09:57:51+08:00",
  "end": "2024-12-22T10:01:05+08:00",
  "via": [],
  "max_step_m": 7
}
```

仅应用人工几何修正：

```bash
python3 scripts/process_route_osm.py --geometry-edits-only
```

### 3. 将车辆轨迹吸附到真实道路

道路匹配属于离线预处理，配置入口是 `road_corridors`。OSM/OSRM 结果进入本地缓存，浏览器只读取最终路线，不在播放过程中请求路线服务。

默认车辆路线优先使用 OSRM。对 OSRM 容易绕远或失败的区域，可在走廊中设置 `force_local: true`，使用缓存的 OSM 道路网做本地最短路匹配。`max_snap_m` 限制原始锚点离道路的最大距离，`max_route_m` 和 `max_route_ratio` 用于拒绝不合理的绕路结果。

新增或调整走廊后，先准备本地 OSM 道路缓存：

```bash
venv-mv/bin/python scripts/process_route_osm.py --prepare-road-corridors
```

需要重新下载缓存时追加 `--refresh-osm`。下载完成后再执行完整处理：

```bash
venv-mv/bin/python scripts/process_route_osm.py --no-driving-match
```

`geometry_edits` 表示人工剪辑意图，优先级高于自动道路匹配。

### 4. 固定区域镜头

使用 `camera_directives`：

```json
{
  "id": "mountain-road-stable",
  "start": "2024-12-22T13:20:00+08:00",
  "end": "2024-12-22T13:55:00+08:00",
  "mode": "fixed_region",
  "center": [90.75, 29.15],
  "zoom": 13.8,
  "pitch": 0,
  "bearing": 0
}
```

车辆可以在区域内移动，镜头不必保持车辆居中。

### 5. 镜头慢于人物或车辆移动

```json
{
  "id": "city-soft-follow",
  "start": "2024-12-22T09:10:00+08:00",
  "end": "2024-12-22T10:27:37+08:00",
  "mode": "soft_follow",
  "zoom": 16.2,
  "follow_factor": 0.025
}
```

`follow_factor` 越小，镜头越稳定、主体越可以偏离中心；取值范围为 0–1。

### 6. 面向雪山或目标点

`look_at` 使用目标 GPS，镜头朝向由 `CameraPolicy` 计算，不依赖人物行进方向：

```json
{
  "id": "kulagangri-facing",
  "start": "2024-12-23T08:19:00+08:00",
  "end": "2024-12-23T12:49:59+08:00",
  "mode": "soft_follow",
  "zoom": 14.8,
  "pitch": 48,
  "look_at": [90.62, 28.18],
  "follow_factor": 0.02
}
```

如果需要严格北向，使用 `bearing: 0`，不要设置 `look_at`。

## 数据流

```text
GPX ────────────────► source/raw-track.json
 │                              │
 │        route-overrides.json  ├──► editorial/exclusions.json
 │                   │          └──► editorial/anchors.json
 │                   ▼
 └── OSM/OSRM ─► navigation/matched-route.json
                             │
                             ▼
                 processed/route-data.js
                    │
          ┌─────────┴─────────┐
          ▼                   ▼
      RouteModel       DirectiveEngine
          │                   │
          └──── CameraPolicy ─┤
                              ▼
                     map-animation.js
                              │
                              ▼
                 MapLibre + MediaOverlay
```

`--geometry-edits-only` 固定从 `navigation/matched-route.json` 重新生成最终层，不在旧的播放路线之上重复修正，因此可以安全重复执行。

## 地点媒体与记忆匣

- `route-overrides.json > media_events` 决定触发坐标、标题、重要级别和视频列表。
- 单段旧事件继续支持 `video + poster`；新事件优先使用 `clips[]`，同一地点聚合多段影像。
- `assets/modules/media-overlay.js` 只管理卡片、切换、延迟加载、暂停和恢复，不参与路线定位。
- `video/web/` 只保存地图用的 H.264 轻量片段与 JPEG 封面；原片仍留在 `video/`。
- `scripts/build_map_video_previews.sh` 是精选片段转码清单，避免批量复制全部 4K 原片。
- `tools/media-card-preview.html` 是独立的媒体卡预览器，便于不用等待轨迹就检查某个事件。

## 道路修复与跨日连接

- 已审核道路几何集中保存于 `route_data/editorial/manual-paths.json`，配置通过 `geometry_edits[].path_id` 引用。
- `scripts/build_editorial_road_paths.py --ids ...` 只更新指定路径并合并旧结果；同时检查 OSRM 端点吸附距离，防止珠峰景区一类端点被吸到十公里外。
- 公共 OSRM 路网缺失时，可在 `road_corridors` 配置本地 OSM 道路缓存；强制本地走廊在没有对应 OSRM 缓存时仍然生效。
- `route_connections` 只作为地图背景接驳线，不进入 `RouteModel`，适合住宿点到次日出发点，不会制造跨夜行驶或虚假的中间时间。

## 修改后的检查顺序

1. 运行对应的路线处理命令。
2. 执行 `node scripts/validate_route_demo.js`。
3. 用真实浏览器检查目标时间段。
4. 确认没有改变目标区间之外的轨迹点、素材时间和镜头规则。
